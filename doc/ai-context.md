# apex-sert — AI Assistant Context

apex-sert is an Oracle APEX security-testing application installed once per database instance, capable of evaluating any APEX application for security weaknesses. Developers access it via the APEX Builder Extension from their own workspace.

---

## Architecture

### Schemas

- **`sert_core`** — all business logic, tables, views, packages, seed data
- **`sert_pub`** — public API surface: synonyms (packages only) and views (`*_pub_v`) wrapping `sert_core` objects; no tables
- **Guardian** uses the same `sert_core` schema within the `product/guardian/` file tree

When suggesting changes, target `sert_core` for logic and `sert_pub` for new public API surface.

### APEX Applications

| App ID | Name | Purpose |
|--------|------|---------|
| 2100 | SERT | Main security evaluation UI |
| 2101 | SERT Admin | Configuration, rule sets, preferences |
| 2102 | Guardian | Optional analytics dashboard over star schema |

Guardian install is controlled by `guardian_switch=runtime` in `sert.properties`.

### Install System (Liquibase)

```
controller.xml
  └── product/sertController.xml
        ├── pre_install (schema/privilege setup)
        ├── product/sert/sert_core/schema.xml   (uses <includeAll> per subdirectory)
        ├── product/sert/sert_pub/schema.xml
        ├── APEX export (f2100, f2101)
        └── post_install
  └── product/guardianController.xml  (only when guardian_switch=runtime)
```

`schema.xml` uses `<includeAll path="..."/>` — **no manual changeset registration is needed** when adding new `.sql` files to a tracked directory.

| Install Script | Use |
|----------------|-----|
| `install.sql` | Production |
| `install_nonprod.sql` | Dev/test |
| `install_no_apex.sql` | Database-only |

### Build / Versioning

- Script: `development/scripts/build_release.sh <VERSION>` → `sert_<VERSION>.zip`
- Versioning: `APEX_VERSION.RELEASE.BUILD` — e.g. `24.2.23` = APEX 24.2, build 23

## File Layout

### sert_core

```
product/sert/sert_core/
  table/              → CREATE TABLE DDL                {table_name}.sql
  constraint_fk/      → FK constraints                  {table_name}.sql
  trigger/            → BU audit triggers               bu_{table_name}.sql
  view/               → Internal views (_v)             {table_name}_v.sql
                        Public views (_pub_v)            {table_name}_pub_v.sql
  package_spec/       → Package specifications          {name}.pks.sql
  package_body/       → Package bodies                  {name}.pkb.sql
  seed_data/          → Reference + rule data           (numbered 010–130)
  index/              → Table indexes
  type/               → PL/SQL types
  procedure/          → Standalone procedures
  function/           → Standalone functions
  job/                → DBMS_Scheduler job definitions
  pre_install/        → Privilege/role setup
  post_install/       → Post-install steps
```

### sert_pub

```
product/sert/sert_pub/
  synonym/            → Package synonyms ONLY (not tables)
  grant/              → EXECUTE grants (packages) and SELECT grants (views)
  view/               → Public views (_pub_v) wrapping sert_core objects
```

`sert_pub` synonyms are for packages only. External table access is always via `_pub_v` views.

### Seed Data Numbering

Files run in alphabetical order by Liquibase `<includeAll>`:

| File | Contents |
|------|----------|
| `010_rule_severity_merge.sql` | Severity levels (HIGH, MEDIUM, LOW) |
| `020_rule_set_types_merge.sql` | Rule set type definitions |
| `030_risks_merge.sql` | OWASP risk codes and names |
| `040_categories_merge.sql` | Security categories |
| `050_prefs_merge.sql` | Public preferences |
| `060_reserved_strings_merge.sql` | APEX substitution strings to exclude |
| `070_rule_criteria_types_merge.sql` | Criteria type keys (SQLI, XSS_ITEM_SYNTAX, etc.) |
| `080_rule_criteria_merge.sql` | SQL evaluation patterns (uses `:l_source` bind) |
| `100_builder_urls_merge.sql` | APEX builder deeplink URL patterns |
| `110_rule_import.sql` | **Security rules** — `data_api.import_rule()` calls |
| `120_rule_set_merge.sql` | Rule set definitions by APEX version |
| `130_create_rule_set_rules.sql` | Rule → rule set membership |

### Guardian

```
product/guardian/sert_core/
  final_fact/         → Fact table DDL (_f suffix)
  staging_table/      → Staging table DDL (_fs suffix, NOLOGGING)
  final_dimension/    → Dimension DDL (_d suffix)
  package_spec/       → .sql (NOT .pks.sql — different from sert_core)
  package_body/       → .sql (NOT .pkb.sql)
```

## Data Model

### Naming Conventions

| Convention | Rule | Example |
|-----------|------|---------|
| PK column | `{table}_id`, `GENERATED BY DEFAULT AS IDENTITY (START WITH 1 NOCACHE ORDER)` | `rule_id`, `eval_id` |
| Business key | `{table}_key` | `rule_key`, `rule_set_key` |
| Boolean flags | `_yn` suffix, `DEFAULT ON NULL 'Y'/'N'`, CHECK `IN ('N','Y')` | `active_yn`, `internal_yn` |
| Internal view | `{table}_v` | `rules_v` |
| Public view | `{table}_pub_v` in both `sert_core` and `sert_pub` | `rules_pub_v` |
| PK constraint | `{table}_pk` | `rules_pk` |
| UK constraint | `{table}_un` | `rules_un` |
| Bool check constraint | `{table}_{column}` | `rules_active_yn` |
| FK constraint | `{referenced_table}_fk` | `rules_fk` |
| BU trigger | `bu_{table}` | `bu_rules` |

### Audit Columns (all sert_core tables)

```sql
created_by  VARCHAR2(250) DEFAULT ON NULL coalesce(sys_context('APEX$SESSION','APP_USER'), user) NOT NULL,
created_on  TIMESTAMP WITH LOCAL TIME ZONE DEFAULT ON NULL systimestamp NOT NULL,
updated_by  VARCHAR2(250) DEFAULT ON NULL coalesce(sys_context('APEX$SESSION','APP_USER'), user),
updated_on  TIMESTAMP WITH LOCAL TIME ZONE DEFAULT ON NULL systimestamp
```

BU trigger sets `updated_by` and `updated_on` only — never `created_*`.

**Guardian tables are different:** no audit columns, no `_yn` flags — use `is_active char(1)`.

### Key Tables

| Table | Purpose |
|-------|---------|
| `rules` | Security rule definitions; `rule_key`+`apex_version` UK; `rule_type` = APEX_VIEW or CUSTOM_QUERY |
| `rule_sets` / `rule_set_rules` | Rule groupings by APEX version |
| `rule_criteria` / `rule_criteria_types` | SQL patterns using `:l_source` bind for code-level checks |
| `categories` | Security categories (AUTHENTICATION, AUTHORIZATION, SQL_INJECTION, XSS, MISCONFIGURATION, ACCESS_CONTROL) |
| `risks` | OWASP risk mappings (risk_code e.g. A05-2021) |
| `rule_severity` | HIGH, MEDIUM, LOW |
| `evals` | Evaluation runs (workspace_id, application_id, rule_set_id, score, pending_score, approved_score) |
| `eval_results` | Per-rule results with JSON result column; page/region/item context |
| `eval_results_history` / `eval_history` | Audit trail |
| `exceptions` | Exception workflow: PENDING → APPROVED / REJECTED; includes `exception_score` for AI scoring |
| `comments` | User comments per finding with full context columns |
| `prefs` | System preferences |
| `logs` | Application logging |
| `builder_urls` | APEX builder deeplink URL patterns by key |

## PL/SQL Packages

Package specs: `package_spec/{name}.pks.sql` · Package bodies: `package_body/{name}.pkb.sql`

### `eval_pkg` — Evaluation Engine
```
eval(p_application_id, p_page_id, p_eval_id_out, p_run_in_background)
process_rules(p_application_id, p_page_id, p_eval_id, p_rule_set_id)
calc_score(p_eval_id)
eval_criteria(p_column_to_evaluate CLOB, p_rule_criteria_type_key, p_application_id) RETURN VARCHAR2
  → executes rule_criteria SQL with :l_source bind
apex_link(p_eval_result_id, p_builder_session_id) RETURN VARCHAR2
delete_eval(p_eval_id)
```

### `data_api` — Rule Import/Export
```
import_rule(p_rule_name, p_rule_key, p_category_name, p_category_key,
    p_risk_code, p_risk_name, p_apex_version, p_help_url,
    p_builder_url_key, p_impact, p_active_yn, p_internal_yn,
    p_rule_type,           -- 'APEX_VIEW' or 'CUSTOM_QUERY'
    p_view_name, p_column_to_evaluate,
    p_component_id, p_component_name, p_column_name, p_item_name, p_shared_comp_name,
    p_operand,             -- EQUALS, IS_NOT_NULL, IS_NULL, CRITERIA, NOT_EQUALS, etc.
    p_val_char, p_val_number, p_case_sensitive_yn,
    p_rule_criteria_type_key,   -- used when p_operand = 'CRITERIA'
    p_additional_where, p_custom_query,
    p_info CLOB, p_fix CLOB, p_time_to_fix,
    p_valid_exceptions, p_rule_severity_name, p_rule_severity_key, p_description)
export_rules_script() RETURN CLOB
export_rules_hex_script() RETURN CLOB    -- hex-compressed, ~90% smaller
import_rules(p_rules BLOB)
create_rule_set_rules(p_rule_set_key, p_apex_version)
```

### `exceptions_api` — Exception Lifecycle
```
add_exception / bulk_add_exception / withdraw_exception / resubmit_stale_exception

upload_exceptions_wizard / download_exceptions

```

## Liquibase Conventions

Every `.sql` file must begin with:

```sql
--liquibase formatted sql
-------------------------------------------------------------------------------
-- Licensed under the Universal Permissive License v 1.0 as shown
-- at https://oss.oracle.com/licenses/upl/
--------------------------------------------------------------------------------
```

Canonical copyright block: `development/licence_snip.txt`

### Changeset Attributes

| Attribute | Rule |
|-----------|------|
| `endDelimiter` | `:;` for DDL · `:/` for PL/SQL |
| `runOnChange` | Always explicit |
| `runAlways` | Always `false` |
| `rollbackEndDelimiter` | Must match `endDelimiter` |
| **`stripComments:false`** | **MANDATORY on all package changesets.** Omitting it causes Liquibase to silently corrupt `q'!..!'` and `q'"..."'` quoted literals inside package bodies. |

### Changeset ID Format

```
{author}:create_{object_type}_{schema}.{object_name}_{epoch_ms}
```

Seed data uses a different convention: `sert:import_sert_rule_{rule-key}-{apex-version}` and `{author}:080_rule_criteria_merge_{KEY}`

### Rollback Clauses

| Object | Rollback |
|--------|---------|
| CREATE TABLE | `--rollback drop table sert_core.{name};` |
| ADD CONSTRAINT | `--rollback alter table sert_core.{name} drop constraint {NAME};` |
| Package spec | `--rollback drop package sert_core.{name};` |
| Package body, trigger, view, seed data | `--rollback not required` |

### Preconditions

Required on CREATE TABLE, ADD CONSTRAINT, ADD COLUMN:

```sql
--preconditions onFail:MARK_RAN onError:HALT
--precondition-sql-check expectedResult:0 select count(1) from all_tables where owner = 'SERT_CORE' and table_name = 'MY_TABLE';
```

Guardian changesets use `dba_tables` / `dba_constraints` / `dba_indexes` (not `all_*`).

### Special Cases

- Files with `&` in content (regex patterns) need `set define {;` before the first changeset
- `<includeAll>` means no manual registration needed for new files in tracked directories

## Guardian ETL Conventions

### Table Naming

- Fact: `sg_*_f` · Staging: `sg_*_fs` · Dimension: `sg_*_d`
- All objects prefixed `sg_`

### ETL Task Contract (non-negotiable)

Every procedure in `sg_etl_task` must:

```sql
procedure load_sg_my_fact_f (p_task_log_id in number) is
    l_rows_changed number := 0;
begin
    -- DML (MERGE / INSERT / DELETE)
    l_rows_changed := sql%rowcount;
    sg_etl_controller.set_task_completed(
        p_task_log_id    => p_task_log_id,
        p_rows_changed   => l_rows_changed,
        p_status_message => apex_string.format('%0 rows loaded to SG_MY_FACT_F', l_rows_changed)
    );
    -- NEVER COMMIT
end load_sg_my_fact_f;
```

1. First parameter: `p_task_log_id in number`
2. Call `sg_etl_controller.set_task_completed(...)` exactly once
3. Never `COMMIT`
4. Assign `l_rows_changed := sql%rowcount` after DML

### Pipeline Registration

`product/guardian/sert_core/seed_data/seed_main_etl_inc.sql` — task IDs increment by 10; `is_active` uses `1`/`0` not `'Y'`/`'N'`.
