# APEX Export via SQLcl — Design Reference

## Purpose

`development/scripts/export_apex.sh` exports one or more APEX applications from the database into the source tree as split Liquibase changelogs. It is intended to be run by a developer before committing, replacing the previous app directory contents so that deleted components are not left behind in source control.

---

## Prerequisites

| Requirement | Notes |
|---|---|
| `sql` (SQLcl) in PATH | Oracle SQLcl CLI |
| `jq` | JSON config parsing — `brew install jq` |
| `$SQLC_CONN` env var | Name of a saved connection in the Oracle SQL Developer VS Code extension (shared store at `~/.dbtools/connections`) |

SQLcl saved connections are managed via the Oracle SQL Developer VS Code extension or from the SQLcl CLI:
```bash
sql -name my_conn          # connect interactively
sql -list                  # list saved connection names
```

---

## Usage

```bash
export SQLC_CONN=my_dev_db
./development/scripts/export_apex.sh 2100
./development/scripts/export_apex.sh 2100 2101 2102   # multiple apps
./development/scripts/export_apex.sh -l 2100          # include licence header in every .sql file
./development/scripts/export_apex.sh --licence 2100 2101
```

### Flags

| Flag              | Default | Effect                                                                  |
|-------------------|---------|-------------------------------------------------------------------------|
| `-l`, `--licence` | off     | Prepends `development/licence_snip.txt` to every exported `.sql` file   |

## Configuration — `apex_config.json`

Located at the repo root. A JSON array where each record maps application IDs to export paths.

### Record types

| Record type | Key fields | Meaning |
|---|---|---|
| Single app | `"id": "2102"` | Exact application ID |
| Range | `"id-lo": "1000", "id-hi": "1999"` | All app IDs in the inclusive range |
| Default | `"id": "default"` | Fallback values for any field not set by the matched record |

### Fields

| Field | Type | Purpose |
|---|---|---|
| `extract_to_path` | string | Repo-relative path passed as `-dir` to SQLcl; the split export lands under `{extract_to_path}/f{app_id}/` |
| `controller_path` | string | Liquibase controller XML for this app group (informational — not used by the script) |
| `preserveInstallXML` | boolean | When `true`, the custom `f{app_id}/install.xml` is backed up before export and restored afterward |

### Example

```json
[
  {
    "id-lo": "1000",
    "id-hi": "1999",
    "extract_to_path": "nonprod/sert/apex/application",
    "controller_path": "nonprod/sert/apex/apexController.xml"
  },
  {
    "id": "2102",
    "extract_to_path": "product/guardian/apex/application",
    "controller_path": "product/guardian/apex/apexController.xml"
  },
  {
    "id": "default",
    "preserveInstallXML": true
  }
]
```

Config field lookup priority: **exact id → id-lo/id-hi range → default record**.

## Per-application Export Flow

```
for each app_id argument:

  1. Resolve config fields (extract_to_path, preserveInstallXML)

  2. If preserveInstallXML = true AND f{id}/install.xml exists
       → back up install.xml to a mktemp file

  3. Delete  {extract_to_path}/f{app_id}/   (ensures deleted components are removed)

  4. Run SQLcl export:
       lb generate-apex-object
         -applicationid {id}
         -split
         -skipexportdate
         -exporiginalids
         -ovf
         -audittype NULL
         -dir {absolute extract_to_path}

  5. If export failed → restore install.xml backup (if any) → mark failed → continue

  6. Restore install.xml backup (if any)

  7. Delete {extract_to_path}/apex_install.xml  (auto-generated by SQLcl, not wanted in source)

  8. Stamp checksums (see below)
```

---

## Checksum Stamping

After each successful export, `stamp_checksums` runs three phases over all `.sql` files inside `f{app_id}/`:

### Phase 1 — per-file SHA-256

Files are collected via `find … -print0 | sort -z` (null-delimited, deterministic path order). SHA-256 is computed on the raw exported content (before any modification). Results are written to a temp map file as `filepath\tchecksum`.

### Phase 2 — app_checksum

All per-file checksums are concatenated (no separator, in sorted path order) and hashed with SHA-256 to produce a single `app_checksum` that represents the entire application export.

### Phase 3 — prepend headers

Two lines are prepended to every `.sql` file via an atomic write-then-rename (`> file.tmp && mv file.tmp file`):

```sql
prompt app_checksum: 2DE9A600B35DA758AB5065B23A022765E1A26789CE4DD1077A96B8C82547917F
-- file_checksum: 69675E02B08AC671E28D68A4D1F1F1B123D5D3D50AA96339B7F93583593DE52B
```

- `prompt app_checksum:` — SQLcl `prompt` directive; same value in every file for the export run
- `-- file_checksum:` — SQL comment; unique per file, reflects raw exported content

SHA-256 is computed with `sha256sum` (Linux) or `shasum -a 256` (macOS); output is uppercased.

---

## Safety Design

- **Connection is tested before any destructive step.** A `select 'CONNECTION_OK' from dual` is run and both the exit code and output are checked. If the connection fails the script aborts with no files modified.
- `set -uo pipefail` — unbound variables and pipe failures are fatal.
- `install.xml` backup is written to `/tmp` before the directory delete, so it survives regardless of export outcome.
- All apps are attempted; a per-app failure sets a flag and `continue`s to the next. The script exits non-zero at the end if any app failed.
